What's left

Where you areFix version 3 is applied, the checker is clean, and the app answers its question: pick a borrower, see who to contact and for what. Everything on this page is polish and plumbing — nothing here can break the build.
The one technical trick on this pageA canvas-app gallery has no "hover" property on its rows — TemplateFill can't react to the mouse. The standard trick: lay a transparent button over each row. Buttons do have HoverFill and PressedFill, the transparent face lets your labels show through, and its OnSelect passes the click to the gallery. One control, and the whole row tints on hover.
Chips: tap to copy — and the navy ones are controlsDark boxes have a Copy button; every inline chip is tap-to-copy (the ⧉ mark). A navy chip with a ▣ mark like btnRowHover is an element — a control in your Tree view; a plain grey chip is a name or value you paste. Property workflow, same as always: select the ▣ control in the Tree view → property dropdown (top-left of the formula bar) → paste the property name → paste the value → Enter.

How it worksThe button is invisible glass over the whole row: Fill transparent, so the name and RIM labels show through from underneath. Hover the row → the glass tints navy at 6%; press → 12%. Clicking the glass runs Select(Parent), which fires the gallery's OnSelect exactly as before — your selection formula, the highlight, everything unchanged. The selected-row tint from TemplateFill still shows through too, so a selected row stays marked and darkens a touch more under the mouse.
If the labels stopped showingThe button must sit on top (last under galBorrowers in the Tree view — newly inserted controls land there). If you see a grey block hiding the text, the Fill didn't take — it must be RGBA(0, 0, 0, 0), not a color.
Check before moving onPreview (F5) → sweep the mouse down the list: each row tints as you pass, the selected row keeps its highlight, clicking any row still loads the contacts on the right.
P2Name, icon & description — the app's public face

The icon and name are what people see on the SharePoint button's target page and in their Power Apps list — thirty seconds now, looks professional forever:

  1. In Studio: Settings (gear, top bar) → General. You should now see Name, Description, and the icon picker.
  2. Name → confirm it reads IBD Client Contact Repository (rename if it still carries a draft name).
  3. Description → paste: Look up any borrower and see their key contacts by communication purpose, plus the BMO coverage team and office contacts. View-only.
  4. Icon → pick the people / contacts glyph → Background color → paste #1F3B6E. You should now see the preview tile turn navy.
Check before moving onThe preview tile in Settings shows a navy tile, a people icon, and the full app name.
P3Save & publish
  1. Top-right of Studio → Save (the disk icon, or Ctrl+S). You should now see "All changes saved".
  2. Next to it → Publish (the up-arrow icon) → Publish this version. You should now see a confirmation that the version is live.
Save ≠ publishSaving makes a new version only you can see; publishing is what pushes it to everyone with the link. Every future tweak needs both clicks — a saved-but-unpublished change is the classic "but I fixed that already" mystery.
Check before moving onThe Publish dialog shows today's date as the Live version.
P4Share the app with the team

A published app is still invisible to colleagues until it's shared — the homepage link would give them a "you need permission" wall. From the maker portal:

  1. Go to make.powerapps.com → left rail → Apps. You should now see IBD Client Contact Repository in the list.
  2. Hover it → ⋯ → Share. You should now see the share panel.
  3. In Enter a name… add the people or the team's security group who should use it. Leave Co-owner unchecked — they're users, not editors.
  4. Skip the email notification if you prefer — the homepage button (P6) is the announcement.
  5. Share.
SharePoint permissions are NOT auto-grantedThe share panel lists the four SharePoint lists under "Data permissions" but sharing the app does not grant access to them — users also need at least Read on the BMO site (bbkonline.sharepoint.com/sites/BMO). Members of the site already have it. If someone outside the site will use the app, add them to the site's Visitors group first, or they'll see empty galleries and connection errors.
Check before moving onThe share panel lists everyone who should have the app, all as User (not Co-owner).
P5Copy the app's web link
  1. Still in make.powerapps.com → Apps → hover the app → ⋯ → Details. You should now see the app's details page.
  2. On the Details tab, find Web link — a long URL beginning https://apps.powerapps.com/play/….
  3. Click the copy icon next to it. That URL is the app — paste it anywhere, it opens the published version in the browser player.
  4. Paste it somewhere safe for the next step (Notepad is fine). Sanity-check it first: open a new tab, paste, Enter — the app should load.
Why this link and not the Studio URLThe Studio address bar URL opens the editor. The Details-page web link opens the player — read-only, always the latest published version, and safe to hand to anyone you shared with in P4.
Check before moving onThe link opens the app in a fresh browser tab, showing the search screen.
P6Post the button on the SharePoint homepage
  1. Open the BMO site homepage → top right → Edit. You should now see the page switch to edit mode with section outlines.
  2. Hover just below where you want the button (near the top works best) → click the circled + → search Button → click the Button web part. You should now see an empty button drop onto the page with a property pane on the right.
  3. Label → paste 📇 IBD Client Contact Repository
  4. Link → paste the web link from P5.
  5. Alignment → Left (or match the section's style).
  6. Top right → Republish. You should now see the homepage live with the button on it.
Prefer it embedded instead of linked?There's also a Microsoft Power Apps web part that runs the app inside the homepage. It's tempting, but the app is designed for a full window — inside a web part it gets a cramped letterbox, and every homepage visit loads the app for every visitor. The button is the right call: one click, full screen, loads only when wanted.
The final test — do this from a colleague's chairAsk one person you shared with (not you) to open the homepage → click the button → search a borrower → tap an email link. If that works on their account, everything works: app permissions, list permissions, published version, homepage link. 🏁

Owning it from here

The maintenance loopData changes (new borrowers, contacts, matrix rows) appear in the app automatically — it reads the lists live, nothing to redeploy. Only app changes need the P3 double-click: Save, then Publish. New team members: add them in the P4 share panel and make sure they can read the BMO site.
