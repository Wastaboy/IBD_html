Pick borrowers by name — and a sharper look

Two independent parts. Part A answers the team's request inside SharePoint itself: the Contact Communication Matrix's borrower lookup gets inverted — today you pick a RIM and the name rides along; after this, you pick the name and the RIM writes itself. One column edit, no data re-entered. Part B is a light visual pass on the app: cards, air, one font. Properties only, no formulas touched.

10 steps ~25 min SharePoint list + Studio ↩ full build guide
0 of 10 steps done

Before you start

The two parts don't depend on each otherPart A (D1–D4) happens entirely in SharePoint and changes nothing in the app — no publish needed. Part B (U1–U6) happens entirely in Studio and touches only cosmetic properties — no controls added, no formulas edited, so nothing can break. Do them in either order.
How Part A works — flip which side of the lookup showsThe Matrix's Borrower RIM column is a lookup: each row stores a pointer to one Master List row, and merely displays one of that row's columns — today, the RIM, with the name riding along as a read-only extra. Part A flips the display: the lookup shows the name (a searchable picker — type a few letters, pick) and carries the RIM along as the read-only extra. Because the stored pointer never changes, every existing Matrix row switches to names instantly, nothing is re-entered — and a mistyped RIM becomes impossible, since the number is always derived from the picked row.
Chips: tap to copy — and the navy ones are controlsEvery inline chip is tap-to-copy (the ⧉ mark). A navy chip with a ▣ mark like conLeft is an element — a control in your Tree view; a plain grey chip is a name or value you paste. Property workflow, same as always: select the ▣ control in the Tree view → property dropdown (top-left of the formula bar) → paste the property name → paste the value → Enter.

Part A — Pick by name in the Matrix

Four steps in SharePoint — one column edit, not a rebuild. When they're done, nobody entering a Matrix row ever types or hunts a RIM again.

D1Invert the Borrower RIM lookup

Open the BMO site → Contact Communication Matrix. You're editing the existing lookup, not creating anything:

  1. Click the Borrower RIM column header → Column settings → Edit. You should now see the "Edit column" panel with the lookup's settings, source list already set to Master List of Exposures.
  2. "Select a column from the list above" dropdown → change RIM to Borrower — the name column (the renamed Title).
  3. Expand More options → in "Add a column to show each of these additional fields": tick RIM, and untick Borrower (the name now is the main column — carrying it twice is clutter).
  4. Click Save. You should now see three things at once: the Borrower RIM column now shows names on every existing row, a new read-only Borrower RIM: RIM column shows the numbers, and the old Borrower RIM: Borrower column is gone.
Classic skin? Same edit, older clothes
  1. Gear (top right) → List settings → under Columns click Borrower RIM.
  2. "In this column:" dropdown → Borrower.
  3. In "Add a column to show each of these additional fields:" tick RIM, untick Borrower → OK.
Why nothing needs re-enteringA lookup stores a pointer to the Master List row — which column it displays is presentation only. Changing the display from RIM to Borrower rewrites nothing: all existing rows switch to names the moment you Save, and the RIM extra fills itself from the same pointer.
Do NOT rename the column while you're in thereTempting as "Borrower RIM" now sounds wrong for a column full of names — the app's OnSelect formula references 'Borrower RIM' by that exact display name, and renaming it breaks the contact loading until the formula is edited too. Keep the name; it's odd for a moment and safe forever. (If the client insists on renaming, say the word — it's a two-line follow-up with one formula edit.)
Check before moving onEvery existing Matrix row shows a borrower name where the RIM used to be, with the RIM sitting in the new read-only column beside it — and you didn't touch a single row.
D2Enter a row by name
  1. Click + New. You should now see the item form — and Borrower RIM is now a name picker: type a few letters of any borrower → the dropdown narrows as you type → pick.
  2. Pick Contact and Purpose as usual → Save.
  3. You should now see the new row with the borrower's name and its RIM — you never touched a number. (Delete the test row if it was a dummy.)
Check before moving onThe whole point, proven: name in, RIM out — derived, not typed, so it can't be wrong.
D3Put the RIM where the eye lands
  1. Drag the Borrower RIM: RIM column header so it sits right after Borrower RIM. If dragging fights you: column header → Column settings → Move left / Move right.
Check before moving onReading a row left to right now tells the story in order: borrower → RIM → contact → purpose.
D4Confirm the app doesn't care

The app matches Matrix rows with 'Borrower RIM'.Id = varBorrower.ID — the stored pointer, never the displayed text (checked: no formula anywhere reads this lookup's .Value). Flipping the display is invisible to it. Prove it in thirty seconds:

  1. Open the published app → search a borrower → select. You should now see the contacts load exactly as before.
  2. That's it — Part A needs no Studio work and no publish.
Did you already run this page's earlier revision?The first push of this page (same morning) had Part A adding a Borrower lookup to the Borrower Contact Directory instead. If you already created it, it's harmless — keep it as a useful borrower tag on each contact, or remove it: Borrower column header → Column settings → Edit → Delete (its "Borrower: RIM" companion disappears with it).
Check before moving onApp behaves identically; the Matrix now speaks names. Part A done.

Part B — The polish pass

Six steps in Studio. Cosmetic properties only — the row math, formulas and control tree stay exactly as they are.

What this pass deliberately does NOT touchNo text sizes inside the gallery rows — those sit in carefully balanced fixed-anchor math (TemplateSize, Y offsets), and bumping them would cascade. The wins below are all free: corners, shadows, spacing, casing, one font.

Do NOT do this to lblPurposeHeaderIts text is data (ThisItem.Title), and galContacts filters on Purpose.Value = lblPurposeHeader.Text — a case-sensitive comparison. Uppercase it and every purpose group silently empties. U5One font, everywhere, on purpose

Depending on which control flavor each insert used, a canvas app can quietly mix Open Sans and Segoe UI — close enough to miss, different enough to look "off". Ten seconds of multi-select makes the font deliberate:

  1. In the Tree view, click lblAppTitle, then hold Ctrl and click every other label in the app — including the ones nested inside the galleries and containers — plus txtSearch.
  2. With them all selected, property dropdown → Font → paste:
Font.'Segoe UI'
Prefer a touch more character?Font.Lato is the one tasteful alternative on the canvas palette — set it the same way. Whichever you pick: one font for the whole app, never a mix.
Check before moving onSweep the app in Preview: every piece of text — title, rows, card, headings, emails — renders in the same face.
U6Save, publish, sweep

Part B changed the app, so the double-click from the final-build page applies:

  1. Save (Ctrl+S) → "All changes saved".
  2. Publish → Publish this version.
  3. Open the app from the homepage button and do a thirty-second sweep: search a borrower, select, hover the list, tap an email. Nothing structural changed — this is confirmation, not testing.
The finish lineRounded cards on a calm grey, an aligned confident title, quiet section labels, one typeface — and in SharePoint, contacts entered by name with RIMs that write themselves. 🏁
