IBD · Client Contact Repository · Build guide

One search, every contact

A view-only canvas app over the four SharePoint lists: type a RIM or borrower name, get the borrower's card and every operational contact grouped by communication purpose. The Master List of Exposures stays the single source of truth — the app reads, never writes. Every step is self-contained, and every name and formula has a copy button.

Phase 1 — Foundation

A blank tablet app, the four connections, and the container skeleton everything else sits in.

S1Create the app
  1. Go to make.powerapps.com → check you're in the client's environment (top-right) → Create → Blank app → Blank canvas app.
  2. Name: IBD Client Contact Repository · Format: Tablet → Create.
  3. Tree view → double-click Screen1 → rename to scrMain.
  4. Settings (gear) → Display → turn Scale to fit off and Lock aspect ratio off, so the layout resizes with the window. Save (Ctrl+S).

Phase 2 — The search panel

Delegable search over the Master List, and the selection that powers everything on the right.

S4Search box & borrower list

a) The search box. Select conLeft → Insert → Text input → rename txtSearch. Flexible height off. Set the hint:

txtSearch · Placeholder (HintText on classic)

"Search by RIM or borrower name"

b) The results list. Select conLeft → Insert → Vertical gallery (Blank layout) → rename galBorrowers → Flexible height on. Set Items — delegable StartsWith, so it stays fast past 500/2000 rows:

galBorrowers · Items — if RIM is TEXT

Filter('Master List of Exposures', StartsWith(Borrower, txtSearch.Text) || StartsWith(RIM, txtSearch.Text))

galBorrowers · Items — if RIM is a NUMBER

Filter('Master List of Exposures', StartsWith(Borrower, txtSearch.Text) || (IsNumeric(txtSearch.Text) && RIM = Value(txtSearch.Text)))
If Power Apps flags BorrowerThe name column may still be the built-in Title column with its display name unchanged in Power Apps — the formula bar's IntelliSense will offer the right one as you type. Same fix everywhere Borrower appears.

c) The row template. Inside the gallery template add two labels — name on top, RIM under it:

Label 1 (bold) · Text

ThisItem.Borrower

Label 2 (smaller, grey) · Text

"RIM: " & ThisItem.RIM
Check before moving onIn Preview, typing narrows the borrower list as you type — by name and by RIM.
S5Selection loads that borrower's contacts

One paste. It remembers the selected borrower, then pulls only that borrower's matrix rows into a local collection — each row enriched with the full contact record. One server call per selection; the nested galleries later never touch SharePoint.

galBorrowers · OnSelect

Set(varBorrower, ThisItem); Set(varLoading, true); ClearCollect(colMatrix, AddColumns( Filter('Contact Communication Matrix', 'Borrower RIM'.Id = varBorrower.ID), ContactRec, LookUp('Borrower Contact Directory', ID = ThisRecord.Contact.Id) ) ); Set(varLoading, false)
Matrix stores RIM as plain text instead of a lookup?Replace the Filter condition with 'Borrower RIM' = varBorrower.RIM — the rest of the paste stays the same. ('Borrower RIM' here is the Matrix's column, spec-named and unconfirmed — substitute whatever V1 found on that list.)
Why a collectionGrouping contacts by purpose needs nested galleries. If those queried SharePoint per row you'd get delegation warnings and a slow app. colMatrix is local, tiny (one borrower's rows), and instant.
Check before moving onSelect a borrower in Preview, then File → open the Variables pane: varBorrower holds the record and colMatrix has that borrower's matrix rows.
S6"No borrowers found"

  2. Text "No borrowers found", grey, centered.

S9The contacts inside each purpose

Inside the galPurposes template, below lblPurposeHeader: Insert → Vertical gallery (Blank) → rename galContacts. Its Items are the matrix rows for this purpose, active contacts only — pick the variant matching your Active Status column type from V1:

galContacts · Items — if Active Status is YES/NO

Filter(colMatrix, Purpose.Value = lblPurposeHeader.Text && ContactRec.'Active Status')

galContacts · Items — if Active Status is a CHOICE

Filter(colMatrix, Purpose.Value = lblPurposeHeader.Text && ContactRec.'Active Status'.Value = "Active")

Then four labels inside the galContacts template:

lblContactEmail · Text

Phase 4 — Finish & ship

S12Polish — clean state & selection highlight

a) Fresh start on every visit.

scrMain · OnVisible

Set(varBorrower, Blank()); Set(varLoading, false); Clear(colMatrix)

b) Show which borrower is selected.

galBorrowers · TemplateFill

If(ThisItem.ID = varBorrower.ID, RGBA(31, 59, 110, 0.08), RGBA(0, 0, 0, 0))
Theme disciplineNavy RGBA(31,59,110,1) only on the header bar, purpose headings and the highlight; body text RGBA(27,36,54,1); secondary grey RGBA(91,102,120,1). Segoe UI throughout (the default). Resist more color — it's a lookup tool.
S13Test everything

F5 from scrMain and run all eleven. All must pass before publishing:

  • Open with nothing typed → full borrower list left, placeholder line right.
  • Partial borrower name → list narrows as you type.
  • Partial RIM → matching borrowers appear.
  • Gibberish → "No borrowers found".
  • Borrower with matrix rows → card + contacts under navy purpose headings; only purposes with contacts show.
  • Borrower without rows → "No contacts recorded for this borrower".
  • A contact marked inactive in SharePoint → not shown.
  • A contact serving two purposes → appears under both headings.
  • Switch borrowers quickly → the right panel always matches the selection, no stale contacts.
  • App checker (stethoscope) → no delegation warnings on galBorrowers.
  • Add a new purpose in the Purpose Master, link a contact, reselect → the new heading appears with no app change.
Check before moving onEleven for eleven. Any miss → the step that built that piece has the formula to re-check.
S14Publish & share — the victory lap
  1. File → Save → Publish → Publish this version.
  2. Power Apps home → the app's ⋯ menu → Share → add the IBD users or their security group as User (not Co-owner).
  3. Hand the BMO team this guide plus your filled-in V1 checklist, so future maintainers know the app's exact column dependencies.
Sharing the app does not grant list accessUsers also need Read on all four SharePoint lists — grant it on the BMO site. The app is view-only, so Read is all they need; don't grant Edit.
