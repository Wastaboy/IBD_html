The verdict

What this build gets right — a lotThe single-screen layout with auto-layout containers is clean and correct. The borrower card (conBorrowerCard with its AutoHeight name and the limit/exposure/utilization line), the BMO coverage team block, and the borrower office contacts block are all new this round — and every single column reference in them is right: the RSM lookups with .Value, the person column with .DisplayName and .Email, all three office email fields. The search formula (name or numeric RIM) is fully delegable and passes the checker clean. Selection highlight, conditional visibility, tappable mailto emails — all correct.
The problems this round ① The selection formula errors out — inside the LookUp, ThisRecord.Contact doesn't mean what it looks like it means. ThisRecord gets captured by the LookUp's own row (a Directory row, which has no Contact column), so the App checker shows three red errors and colMatrix fails on every tap. Same bug family as fix №2's F3 — name shadowing. ② colMatrix is collected and then thrown away — nothing on the screen displays it. The purposes/contacts galleries from fix №2 are gone from this build, and no replacement exists. The core question of the app — who do I contact, for what — never renders. ③ CustomGallerySample is back in your data sources — the fingerprint of a gallery inserted with a sample layout at some point. ④ Checker hygiene: a performance flag on lblNoResults, two missing accessible labels, six missing tab stops — plus the three cosmetic true-ups from review №1, still pending.
The plan: a flat contact gallery, not the nested oneThe nested purposes-gallery design has burned you twice, and this build quietly dropped it. So this round prescribes the simpler architecture: one flat gallery straight off colMatrix, each row showing purpose · name · job · email · phones, sorted by purpose. One template level, no gallery-in-gallery, same information. (The grouped-headings version still lives in fix №2 if you ever want it back.)
Chips: tap to copy — and the navy ones are controlsDark boxes have a Copy button; every inline chip is tap-to-copy (the ⧉ mark). A navy chip with a ▣ mark like galMatrix is an element — a control in your Tree view; a plain grey chip is a name or value you paste. Property workflow, same as always: select the ▣ control in the Tree view → property dropdown (top-left of the formula bar) → paste the property name → paste the value → Enter.

The seven steps

F1Fix the selection formula — the shadowing bug

Your OnSelect reads LookUp('Borrower Contact Directory', ID = ThisRecord.Contact.Id). Inside a LookUp, ThisRecord means the LookUp's own row — a Directory row. The Directory has no Contact column, hence "Name isn't valid: 'Contact' isn't recognized" and the two follow-on errors. The fix is the same medicine as fix №2's F3: give the outer rows a name with As, then use that name.

Select galBorrowers → property dropdown → OnSelect → select everything → paste:

galBorrowers · OnSelect — the As MatrixRow alias is the fix

Set(varBorrower, ThisItem); Set(varLoading, true); ClearCollect(colMatrix, AddColumns( Filter('Contact Communication Matrix', 'Borrower RIM'.Id = varBorrower.ID) As MatrixRow, ContactRec, LookUp('Borrower Contact Directory', ID = MatrixRow.Contact.Id) ) ); Set(varLoading, false)
Why the alias worksAs MatrixRow names each Matrix row explicitly. Inside the LookUp, bare ID still means the Directory row's ID (innermost scope — that's what you want), and MatrixRow.Contact.Id unambiguously reaches the outer row. No name can get captured by the wrong scope. This is the second time shadowing has bitten — the rule of thumb: the moment one record-scope formula sits inside another, alias the outer one with As.
One yellow flag will remain — that's expectedThe checker will still show a Medium delegation warning on 'Borrower RIM'.Id. That comparison can't be sent to SharePoint, so Power Apps downloads the first 500 rows of the whole Matrix and filters locally. Harmless at your current data size — F7 has the insurance settings and the long-term fix if the Matrix ever grows big.
Check before moving onApp checker (the stethoscope icon, top right) → the three red Formula errors are gone. No red underline in the formula bar.

Why DelayOutput matters hereRight now every keystroke in the search box re-runs the SharePoint filter — type "Northwind" and that's nine queries. DelayOutput makes the box wait until you pause typing (~half a second), then fire once. Free performance, one toggle.

F7Delegation insurance & the re-test

a) Raise the data row limit so the one remaining delegation warning has headroom: Settings (gear, top bar) → General → Data row limit → 2000.

What that warning actually means — and the real fix if you ever need itThe Matrix filter compares 'Borrower RIM'.Id, which SharePoint can't evaluate server-side, so Power Apps pulls the first 2,000 Matrix rows and filters on the device. Fine for hundreds of rows; wrong answers past 2,000 (rows silently missing). If the Matrix ever heads that way: add a plain Number column RIM Number to the Matrix, fill it per row, and filter with RIM Number = varBorrower.RIM — plain number equality delegates fully and the warning disappears.

b) F5 (Preview) and run the list:

  • App checker → zero formula errors; at most the one Medium delegation warning on OnSelect.
  • Search: type a partial borrower name → results filter after a brief pause (that's DelayOutput working). Garbage text → "No borrowers found", centred and grey.
  • Tap the test borrower → loading flash, then contact cards sorted by purpose: amber purpose · bold name · grey job · navy email · phones. Coverage team and office contacts below.
  • A contact tied to two purposes appears twice, once per purpose — by design.
  • An inactive contact appears nowhere. (Flip them Active in SharePoint, reselect the borrower → they appear. Flip back.)
  • Tap a contact email → mail draft opens. Same for RSM / RIM officer / office emails.
  • Borrower with no Matrix rows → "No contacts recorded for this borrower"; card, coverage and office blocks still render.
  • Borrower with one RSM → no dangling "RSM 2 ·" line.
